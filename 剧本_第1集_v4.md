**子问题3：IK 是什么——为什么 G1 上用不了**

GraspNet 给了末端目标位置，但机械臂执行动作靠的是关节角度，不是位置。从位置反推关节角，这叫逆运动学，IK。

我们用一个纯 Python 的二连杆机械臂来演示 IK 的原理，不需要仿真：

【终端】

```bash
cd ~/g1_o6_grasp_demo
python3 ik_demo.py
```

【可视化窗口弹出，展示五种情况】

大家看这个图——

左边几个图，给定不同的目标点，IK 自动算出来两个关节角应该是多少度，机械臂末端准确地到达目标。终端里打印出来的验证误差是 0，说明数学上完全正确。

最右边那个图，目标点超出了臂展范围，IK 无解，直接告诉你不可达。

（停顿）

【重点】但是 IK 数学上正确，不代表真实机器人能执行到位。

G1 机械臂每个关节都有力矩上限。IK 算出"关节要转到某个角度"，但关节的电机力矩不够，在规定时间内根本转不到那个角度就停了。这就是为什么我们在 G1 上用 `servo_drive` 跑 IK，十次测试成功零次。

最终方案是 `teleport_arm`——直接把关节角写进 MuJoCo，跳过力矩动力学，十次成功十次。

**子问题3补充：坐标系变换是什么意思**

GraspNet 输出的坐标，是相机坐标系下的。机械臂执行用的是 G1 基座坐标系。两个坐标系原点不同、朝向不同。我们来看变换前后的坐标差异：

【终端】

```bash
python3 -c "
import numpy as np

# 模拟 GraspNet 输出的一个抓取位置（相机坐标系）
pos_camera = np.array([0.2, -0.1, 0.8])  # 单位：米
print('GraspNet 输出（相机坐标系）：', pos_camera)

# 模拟相机外参矩阵（相机坐标系 → MuJoCo 世界坐标系）
# 注意：OpenCV 坐标系 Z朝前Y朝下，MuJoCo Z朝上，有90度旋转差
R_cam2world = np.array([
    [ 0,  0,  1],   # MuJoCo X = 相机 Z（朝前）
    [-1,  0,  0],   # MuJoCo Y = -相机 X
    [ 0, -1,  0],   # MuJoCo Z = -相机 Y（朝上）
])
t_cam2world = np.array([0.5, 0.0, 1.2])  # 相机在世界坐标系里的位置

pos_world = R_cam2world @ pos_camera + t_cam2world
print('坐标系变换后（世界坐标系）：', np.round(pos_world, 3))

# 再从世界坐标系变换到 G1 基座坐标系
t_world2base = np.array([0.3, 0.0, 0.0])
pos_base = pos_world - t_world2base
print('最终（G1基座坐标系）：    ', np.round(pos_base, 3))

print()
print('三个坐标系下，同一个点的坐标完全不同。')
print('如果不做变换，直接把相机坐标传给机械臂，手臂会跑到错误位置。')
"
```

看终端输出——同一个物理位置，在三个坐标系下的数值完全不同。这就是为什么坐标系变换不能省。

## ▌二、三层架构（2 分钟）

【云文档】滚到"二、系统架构"架构图

三层架构，数据从上往下流：

```
感知层：点云采集 → GraspNet 推理 → 输出 .npz
    ↓  6D 抓取位姿（相机坐标系）
规划层：坐标系变换 → 关节角规划 → 输出关节角数组
    ↓  7-DOF 关节角
执行层：teleport_arm 写入关节角 → 夹爪控制 → 抓取完成
```

每层通过文件传数据，落盘的好处是每层可以独立调试。
历史运行的 200+ 条数据就在 `g1_graspnet_results_closest_group/` 里，随时可以拿来回放。
