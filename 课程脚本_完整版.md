## 第一部分：系统是什么（10 分钟）

### 1.1 开场

【云文档】顶部标题

这门课做一件具体的事——把 GraspNet 这个学术抓取网络，移植到宇树 G1 人形机器人上，在 MuJoCo 仿真里跑通完整的抓取流程。

G1 是宇树机器人的人形机器人，双臂各七个自由度。MuJoCo 是目前强化学习和机器人仿真里用得最广的物理引擎，我们用它模拟 G1，不碰真实硬件。

机器人抓取这件事，拆开来是三个子问题：

- **感知**：物体在哪，怎么抓最稳
- **规划**：从目标位置反推出各关节要转多少度
- **执行**：控制关节运动到目标角度

这门课从头到尾讲的，就是这三件事在代码里是怎么实现的。

### 1.4 IK 是什么——为什么 G1 上用了特殊方案

GraspNet 给了末端目标位置，但机械臂执行靠的是关节角度，不是位置。从位置反推关节角，这叫逆运动学，IK。

我们用一个纯 Python 的平面二连杆机械臂来演示 IK 的原理，不需要仿真，不需要 GPU：

【终端】

```bash
cd ~/g1_o6_grasp_demo
python3 ik_demo.py
```

终端会打印：

```
==================================================
  逆运动学演示  大臂L1=1.0m  小臂L2=0.8m  最大臂展=1.8m
==================================================

目标点 ( 1.2,  0.8)：
  theta1 =  ...  deg  (肩关节)
  theta2 =  ...  deg  (肘关节)
  验证误差 = 0.00000000m

目标点 ( 2.5,  0.0)：
  目标点 (2.5, 0.0) 不可达（距离 2.500 > 最大臂展 1.800）
```

同时弹出一个五格图。

【仿真/图像窗口弹出】

看这五个格子——

- 前四个图：给定不同目标点，IK 自动算出肩关节和肘关节应该转多少度，机械臂末端准确到达目标。终端里打印的验证误差是 0，说明数学上完全正确。

- 最后一个图（目标点 2.5, 0.0）：这个点的距离是 2.5 米，超出了最大臂展 1.8 米，IK 无解，程序直接告诉你不可达，格子背景变红。

【重点】但是 IK 数学上正确，不代表真实机器人能执行到位。

G1 机械臂每个关节都有力矩限制。IK 算出"关节要转到某个角度"，但关节电机的力矩不够，在规定时间内可能转不到位就停了。这就是为什么我们在 G1 上用 servo_drive 方式跑 IK 接近全程——力矩不够，最后几毫米差一点，夹爪就夹不住箱子。

最终方案是在接触阶段用 `teleport_arm`——直接把关节角写进 MuJoCo，跳过力矩动力学，保证手指精确到位。这个设计思路后面会详细讲。

### 1.5 坐标系变换

GraspNet 输出的坐标是相机坐标系下的。机械臂执行用的是 G1 基座坐标系。两个坐标系原点不同、朝向不同，必须做变换。

【终端】

```bash
python3 -c "
import numpy as np

# 模拟 GraspNet 输出的抓取位置（相机坐标系，Z 轴朝前）
pos_camera = np.array([0.2, -0.1, 0.6])
print('GraspNet 输出（相机坐标系）：', pos_camera)

# 相机坐标系到 MuJoCo 世界坐标系的旋转
# MuJoCo: X朝前、Z朝上；OpenCV相机: Z朝前、Y朝下
R = np.array([
    [ 0,  0,  1],   # MuJoCo X = 相机 Z
    [-1,  0,  0],   # MuJoCo Y = -相机 X
    [ 0, -1,  0],   # MuJoCo Z = -相机 Y
])
t_cam_in_world = np.array([0.5, 0.0, 1.2])  # 相机在世界坐标系中的位置

pos_world = R @ pos_camera + t_cam_in_world
print('变换后（世界坐标系）：    ', np.round(pos_world, 3))

# 世界坐标系到 G1 基座坐标系
t_base_in_world = np.array([0.3, 0.0, 0.0])
pos_base = pos_world - t_base_in_world
print('最终（G1 基座坐标系）：   ', np.round(pos_base, 3))

print()
print('同一个物理位置，在三个坐标系下数值完全不同。')
print('不做变换直接传给机械臂，手臂会跑到错误位置。')
"
```

看输出——同一个物理位置，在三个坐标系下的数字完全不同。这就是为什么代码里有坐标系变换这一步，不能省。

### 1.6 三层架构总览

【云文档】架构图

整个系统三层，数据从上往下流：

```
感知层：相机图像 → 点云生成 → GraspNet 推理 → 抓取方向（TOP/SIDE）+ 置信度
           ↓  最优抓取的位置（世界坐标系）
规划层：坐标系变换 → IK 求解 → approach 关节角组 + grasp 关节角组
           ↓  关节角字典
执行层：servo_drive 平滑运动 → teleport_arm 精确接触 → 夹爪闭合 → 举起
```

每层通过字典或文件传数据，每层可以独立调试。
历史运行的结果文件在 `g1_graspnet_results_closest_group/` 里，随时可以拿来复查。

### 3.2 grasp_data_bridge.py

【终端】

```bash
code ~/g1_o6_grasp_demo/manipulator_grasp/grasp_data_bridge.py
```

这个文件只有一个类 `GraspDataBridge`，核心是第 10 行的 `extract_grasp_data` 函数。

它做的事情：拿到 GraspNet 预测系统的原始输出，提取出标准格式数据。

看函数开头定义的这个字典（第 11～19 行）：

```python
data = {
    'target_position': None,
    'target_rotation': None,
    'estimated_object_position': None,
    'estimated_object_size': None,
    'is_top_grasp': False,
    'grasp_params': None,
    'success': False
}
```

这就是这个函数的输出格式。无论 GraspNet 给了什么，最终都整理成这个字典。`success` 字段很关键——如果提取失败，`success` 是 `False`，下游代码检查这个字段决定要不要继续执行。

再看第 70～77 行，这里有一个细节：

```python
u, s, vh = np.linalg.svd(data['target_rotation'])
data['target_rotation'] = u @ vh
if np.linalg.det(data['target_rotation']) < 0:
    ...
```

GraspNet 输出的旋转矩阵在数值计算中可能有微小误差，不严格正交。这里用 SVD 分解重新构造最近的正交矩阵，保证后续计算不出数值问题。

【重点】这种"防御性编程"在工程代码里很常见——不相信上游给的数据一定是干净的，自己做一道校验。

**servo_drive（第 66 行）——平滑运动**

```python
def servo_drive(env, target_params: dict, steps: int, sleep=True):
    """5 阶多项式平滑插值 ctrl，从当前值驱动到 target_params。"""
    start_ctrl = {}
    for jname in target_params:
        aid = mujoco.mj_name2id(env.mj_model, mujoco.mjtObj.mjOBJ_ACTUATOR, jname)
        if aid != -1:
            start_ctrl[jname] = float(env.mj_data.ctrl[aid])

    for step in range(steps):
        t = step / max(steps - 1, 1)
        alpha = 6*t**5 - 15*t**4 + 10*t**3       # ← 5阶多项式
        for jname, end_val in target_params.items():
            aid = mujoco.mj_name2id(...)
            if aid != -1:
                env.mj_data.ctrl[aid] = start_ctrl[jname] + alpha * (end_val - start_ctrl[jname])
        env.step()
```

【重点】这里用的是 5 阶多项式，公式 `6t⁵ - 15t⁴ + 10t³`。

为什么不用线性插值？线性插值在起点和终点速度突变，机械臂有抖动。5 阶多项式保证起止速度和加速度都是零——从静止平滑加速，再平滑减速到静止，视觉上非常流畅。

注意这里写的是 `ctrl`，走的是 MuJoCo 执行器通道，有物理约束，关节力矩不够时会停不到位。

---

**teleport_arm（第 86 行）——精确定位**

```python
def teleport_arm(env, joint_targets: dict):
    """瞬移：同时设置 qpos 和 ctrl，保证手指精确接触箱子。"""
    for jname, val in joint_targets.items():
        jid = mujoco.mj_name2id(env.mj_model, mujoco.mjtObj.mjOBJ_JOINT, jname)
        if jid != -1:
            env.mj_data.qpos[int(env.mj_model.jnt_qposadr[jid])] = val   # 直接写关节角
        aid = mujoco.mj_name2id(env.mj_model, mujoco.mjtObj.mjOBJ_ACTUATOR, jname)
        if aid != -1:
            env.mj_data.ctrl[aid] = val
    mujoco.mj_forward(env.mj_model, env.mj_data)
```

和 `servo_drive` 的区别——这里直接写 `qpos`，跳过力矩动力学，手臂瞬间到位。

**完整执行序列（第 270 行）**

【重点】这是整个项目最值得看的部分：

```python
# [4] Servo 平滑驱动到 approach 位置（视觉上流畅）
servo_drive(env, approach_params, APPROACH_STEPS)   # 1500 步，5阶多项式

# [5] Teleport 到 grasp 位置（直接设 qpos+ctrl，确保手指精确接触箱子）
teleport_arm(env, grasp_params)

# [6] 闭合手指
env.set_hand_close()

# [7] 举起手臂（5 阶多项式平滑，避免急加速把箱子甩掉）
lift_arm(env, LIFT_DELTA, LIFT_STEPS)               # 1500 步
```

这就是整个系统执行层的设计哲学：

- **[4] servo_drive**：手臂从预备姿态运动到物体上方，距离大、需要视觉流畅，用 5 阶多项式，1500 步走完。
- **[5] teleport_arm**：从 approach 精确移到 grasp 位置，这段距离很短（约 2cm）但精度要求高。servo 走力矩通道，G1 力矩限制可能导致最后几毫米差一点点，夹爪就夹不住。teleport 直接写关节角，保证精确到位。
- **[7] 举起也用 5 阶多项式**：注释里专门说了"避免急加速把箱子甩掉"——如果直接线性加速，抬手时的冲击力会让箱子飞出去。

（停顿）

**Servo → Teleport 混合策略不是妥协，是有意识的工程设计**：大范围移动用 servo 保流畅，精确接触用 teleport 保精度。两个工具各用其长。

## 第五部分：为什么这样设计（5 分钟）

### 5.1 Servo 和 Teleport 各自的边界

刚才演示里，手臂运动经历了 Servo → Teleport 两个阶段。为什么要混合？用一种不行吗？

**如果全程 Servo**

手臂从初始位置到 approach 再到 grasp，全部走力矩通道。

在手指接触箱子之前，Servo 工作正常。接触的瞬间，MuJoCo 开始计算接触力——手指和箱子之间的物理碰撞响应。

这个接触力会反作用到手臂关节，让实际关节角度偏离 Servo 正在追踪的目标值。偏差通常是几毫米到一厘米。

听起来不多——但箱子才 7cm 宽，手指要精确对准中心。偏 1cm，抓取成功率大幅下降。

**如果全程 Teleport**

直接写 `qpos` 在物理上意味着：两帧之间关节角度突变了若干度，等效速度可能是每秒几百弧度。真实电机的速度上限大概是每秒 10～15 弧度。

大范围 Teleport 在物理上是不可能发生的运动。MuJoCo 虽然不会报警，但会产生极大的"虚假速度"，引起仿真不稳定——轻则箱子被弹飞，重则整个模型状态出错。

**混合策略**

大范围移动（手臂横跨 40cm 从预备姿态运动到箱子上方）：

- 这段没有接触力干扰
- 需要视觉流畅，学生在看
- 用 Servo + 5 阶多项式，完美适合

精确接触（最后 2cm，从 approach 到 grasp）：

- Servo 走到这里会受接触力影响，精度不够
- 位移很小（2cm），Teleport 引起的"虚假速度"在可接受范围
- 用 Teleport，保证精度

【重点】这不是因为 Servo 不好才用 Teleport，而是两个工具有各自擅长的场景。工程上选对工具，才能在精度和稳定性之间取得最佳平衡。
